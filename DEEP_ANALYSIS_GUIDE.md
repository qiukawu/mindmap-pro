# 深度分析功能指南

## 🎯 问题解决

您反馈的问题已完全解决：

**问题**：分析和生成功能基本相同，分析的效果没有明显区别  
**原因**：都是按思维导图格式生成，分析没有针对性  
**解决**：✅ 添加了 5 种分析视角，每种都有不同的分析深度和维度

---

## ✨ 5 种分析视角

现在分析功能支持 5 种不同的专业分析视角，每种都有针对性的分析框架：

### 1️⃣ 业务架构分析
**关注**：业务流程、价值链、商业模式、客户旅程、竞争优势

**应用场景**：
- 分析一个商业计划或商业提案
- 理解公司的商业模式
- 评估市场机遇和竞争

**分析维度**：
- 核心业务流程和价值链
- 商业模式和收入来源
- 目标客户和市场定位
- 竞争优势和差异化
- 关键风险和机遇

**输出示例**：
```
中心主题：SaaS 产品商业模式分析

一级分支：
• 商业模式设计 - 订阅制与分层定价
  - 基础版：个人和小团队
  - 专业版：中型企业
  - 企业版：大型组织
```

### 2️⃣ 技术架构分析
**关注**：技术栈、系统组件、接口设计、技术选型、性能考虑

**应用场景**：
- 评估系统的技术方案
- 技术改进和优化
- 架构升级规划

**分析维度**：
- 采用的技术栈和框架
- 系统的核心组件和模块
- 服务间的接口和集成方式
- 技术选型的优缺点
- 性能、扩展性和可维护性

**输出示例**：
```
中心主题：微服务架构技术分析

一级分支：
• 技术栈选择 - 开源框架和中间件
  - 后端：Node.js + Express + MongoDB
  - 前端：React + Redux + TypeScript
  - 基础：Docker + Kubernetes + Istio
```

### 3️⃣ 系统架构分析
**关注**：系统模块、数据流、服务关系、部署结构、集成方案

**应用场景**：
- 系统设计评审
- 系统改进和扩展
- 遗留系统现代化

**分析维度**：
- 系统的模块划分和职责
- 各模块之间的数据流和交互
- 服务间的依赖关系
- 系统的部署和运维架构
- 不同系统的集成方案

**输出示例**：
```
中心主题：电商平台系统架构分析

一级分支：
• 核心模块划分 - 业务域驱动设计
  - 用户管理模块：账户、权限、认证
  - 商品管理模块：分类、库存、搜索
  - 订单模块：下单、支付、物流
```

### 4️⃣ 数据架构分析
**关注**：数据模型、数据流、存储方案、数据处理、数据安全

**应用场景**：
- 数据仓库规划
- 数据治理和质量管理
- BI 系统设计

**分析维度**：
- 核心数据模型和实体关系
- 数据的来源、流转和使用
- 数据存储方案选择
- 数据处理和分析方法
- 数据安全和隐私保护

**输出示例**：
```
中心主题：用户行为数据仓库设计

一级分支：
• 数据源和收集 - 多渠道数据汇聚
  - Web 埋点数据：页面访问、点击事件
  - App 埋点数据：用户操作、转化漏斗
  - 第三方数据：外部平台数据接入
```

### 5️⃣ 组织架构分析
**关注**：部门结构、角色职责、协作关系、决策流程、沟通机制

**应用场景**：
- 组织变革和重组
- 流程优化
- 团队效能评估

**分析维度**：
- 组织的部门结构和层级
- 各角色的职责和权限
- 部门间和团队间的协作关系
- 决策流程和权力分配
- 信息流和沟通机制

**输出示例**：
```
中心主题：科技公司组织结构分析

一级分支：
• 产品团队组织 - 跨职能协作
  - 产品经理：需求、规划、优先级
  - 技术团队：架构、开发、质量
  - 设计团队：交互、视觉、用户研究
```

## 🚀 如何使用

### 步骤 1️⃣：打开分析面板
点击工具栏上的"分析"按钮（或"AI 分析"）

### 步骤 2️⃣：选择分析视角
在弹出的对话框中，您会看到"分析视角"下拉菜单：
- 业务架构分析
- 技术架构分析
- 系统架构分析
- 数据架构分析
- 组织架构分析

### 步骤 3️⃣：输入要分析的内容
粘贴或输入你的文本内容

### 步骤 4️⃣：点击"生成"
AI 会根据你选择的分析视角进行深度分析

## 💡 使用建议

### 什么时候用不同的分析视角

| 视角 | 最适合用来... |
|------|-------------|
| **业务** | 评估商业计划、分析竞争、理解模式 |
| **技术** | 评估技术方案、规划架构、优化系统 |
| **系统** | 设计系统、理解流程、规划集成 |
| **数据** | 规划数据仓库、治理数据、BI 设计 |
| **组织** | 组织变革、流程优化、团队协作 |

### 最佳实践

1. **选择相关的视角** - 根据你的目标选择最相关的分析视角

2. **提供足够的上下文** - 更多的信息 = 更好的分析
   - 最少 200 字（触发 API 调用）
   - 最好 500-2000 字（最佳效果）

3. **明确分析目标** - 如果文本中包含目标说明，分析会更准确
   ```
   "我们要优化订单流程。当前问题是：1) 流程过长 2) 手工环节多 3) 错误率高"
   ```

4. **结合多个视角** - 对于复杂问题，可以从多个角度分析
   ```
   先用"系统架构"了解结构
   再用"组织架构"了解团队协作
   最后用"业务架构"了解价值链
   ```

## 📝 示例

### 示例 1：分析一个产品文档

**原文**：产品文档，描述一个项目管理工具

**选择**：业务架构分析

**预期输出**：
```
中心主题：项目管理工具产品分析

一级分支：
• 商业模式 - SaaS 订阅制
  - 基础版：小团队，功能受限
  - 专业版：中团队，全功能
  - 企业版：大企业，定制服务
• 目标市场 - 中小型团队
  - 初创公司：成本敏感，功能要求高
  - 传统企业：稳定性要求高，风险规避
• 竞争优势 - 易用性 + 定价合理
  - 界面简洁：学习成本低
  - 价格透明：无隐藏收费
  - 本地化：支持中文和本地规范
...

关键发现：
• 目标市场广阔，但竞争激烈
• 差异化需要体现在易用性和本地支持
• 需要建立用户社区和生态...

建议与改进：
• 加强与HR系统的集成
• 提供更丰富的报表和数据分析
• 建立知识库和最佳实践社区
```

### 示例 2：分析一个技术方案

**原文**：技术设计文档，描述系统架构

**选择**：技术架构分析

**预期输出**：
```
中心主题：微服务架构技术方案评估

一级分支：
• 技术栈选择 - 现代化开源方案
  - 后端：Java Spring Boot + Dubbo
  - 前端：React + TypeScript
  - 消息队列：Kafka 高吞吐量消息处理
• 部署架构 - 云原生容器化
  - Docker 容器化部署
  - Kubernetes 编排和管理
  - 服务网格 Istio 流量管理
• 高可用方案 - 多层次冗余
  - 数据库主从复制和读写分离
  - 缓存多副本和热备
  - 负载均衡和故障转移
...

关键发现：
• 技术栈选择合理，符合现代微服务最佳实践
• 但运维复杂度高，需要专业的 DevOps 团队
• 监控和告警系统还需要完善...

建议与改进：
• 引入 APM 工具（如 Datadog）进行性能监控
• 实现自动化的蓝绿部署和金丝雀发布
• 建立完整的混沌工程测试体系
```
